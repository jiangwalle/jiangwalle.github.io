redux入门教程(三) ~ 异步action

在前面介绍的redux基本用法中，用户发送Action，store调用reducer立即计算出新的state，再发送到view中重新渲染。

现在有一个问题是，如果这个action是异步的，例如，发送query_order这样的action来查询订单，应该怎么处理呢？

有这么几个可能的方案：

  1. View：在view中，先发送一个action如start_query_order，然后执行异步操作调用api，拿到结果后，再发送一个completed_query_order，表示查询结束。理论上其实是可以的，但是view在功能上是负责页面渲染的，异步操作放在这里并不合适。同时，每一个页面都如此实现，会很繁琐。

  2. Reducer：是处理action，并生成新的state的函数。那么是不是可以在reducer里实现上面View中的那套逻辑呢？答案是不可以！因为Reducer在设计上是纯函数，不允许发生异步调用。

  3. 改造dispatch函数，让异步操作在dispatch的过程中完成。

方案3是可行的，但是需要借助一个东西来实现：redux的中间件

中间件概念

中间件是软件设计中一种很重要的思想，它允许使用者自定义插件，去修改软件允许时的数据流。redux里的中间件跟node的web服务框架Express很类似。

Redux异步中间件

redux-thunk

介绍

一次异步操作至少需要两个action。

例如查询订单：

  • 用户点击页面，触发第一个启动action query_order_start。界面可以根据这个action做出相应的变化，比如显示spin图标，表明正在加载。
  • 系统发出第二个结束action query_order_success 或者 query_order_error。在请求完毕之后自动触发。

store.dispatch函数默认情况下只能接受一个action对象作为参数，但是，如果能修改dispatch函数，让它接受一个函数作为参数，这个函数中，先发送一个启动action，然后进行异步操作，操作完成后，再发送结束action。那么异步功能就完成了。这就是redux-thunk中间件的设计思想。

redux-saga

什么是saga

另一个著名的redux中间件，它采用了和上面两个库完全不同的设计理念。它让异步行为成为架构中独立的一层(称为saga)，既不在action creator中，也不和reducer沾边。

它的出发点是把副作用 (Side effect，异步行为就是典型的副作用) 看成”线程”，可以通过普通的action去触发它，当副作用完成时也会触发action作为输出。

具体特点为：

  • redux-saga 完全基于 Es6 的 Generator Function
  • 不使用 actionCreator 策略，而是通过监控 action, 然后在自动做处理
  • 所有带副作用的操作（异步代码， 业务控制逻辑代码）都被放到 saga 中

到底什么是 saga

The term saga is commonly used in discussions of CQRS to refer to a piece of code that coordinates and routes messages between bounded contexts and aggregates.

这个定义的核心就是 CQRS-查询与责任分离 ，对应到 redux-sage 就是 action 与 处理函数的分离。 实际上在 redux-saga 中，一个 saga 就是一个 Generator 函数。

例如：

saga术语解释

  • watcher saga：负责编排和派发任务的 saga
  • worker saga：真正负责处理 action 的函数
  • saga helper：如上面例子中的 takeEvery，简单理解就是用于监控 action 并派发 action 到 worker saga 的辅助函数
  • Effect：redux-saga 完全基于 Generator 构建，saga 逻辑的表达是通过 yield javascript 对象来实现，这些对象就是Effects。这些对象相当于描述任务的规范化数据（任务如执行异步函数，dispatch action 到一个 store），这些数据被发送到redux-saga中间件中执行。

通过这种 effect 的抽象，可以避免 call 和 dispatch 的立即执行，而是描述要执行什么任务，这样的话就很容易对 saga 进行测试，saga 所做的事情就是将这些 effect 编排起来用于描述任务，真正的执行都会放在 middleware 中执行。
